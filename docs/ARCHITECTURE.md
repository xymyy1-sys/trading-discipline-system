# 交易纪律系统架构

## 目标

系统定位是 A 股短线/超短线个人交易纪律工具，不接券商交易 API，不做真实下单。它负责行情读取、资金流观察、信息差跟踪、买入前检查、集中进攻风控、卖出执行卡、交易日志和复盘。

## 模块

- `frontend`: React + TypeScript + Vite，交易终端式页面。
- `backend`: FastAPI，提供业务 API、规则引擎和数据适配层。
- `SQLite`: 第一版本地持久化，保存持仓、交易日志、退出卡、市场快照。
- `MarketDataProvider`: 行情适配层。当前优先直连东方财富板块资金接口；当东方财富 `clist` 类板块接口被服务器网络或风控拦截时，自动切换到新浪财经 `MoneyFlow` 板块资金服务。AkShare 保留为辅助适配层。
- `MainlineClassifier`: 位于行情适配层内，把原始行业/概念/风格板块归因到可交易主线，例如机器人/物理 AI、PCB/玻璃基板、CPO/光模块、创新药、电网设备/电力、贵金属/黄金等。宽泛风格标签只保留在资金证据页，不作为题材雷达主线。
- `InformationDifferential`: 信息差适配与评分。当前聚合东方财富快讯和 CCTV 新闻联播，映射 A 股相关板块，再结合资金流状态判断是否“资金已验证”。

## 数据原则

- 行情、资金流、板块、指数等数据必须通过 API 获取。
- 题材强度必须优先使用板块/概念/行业资金流数据。个股行情只用于板块内部龙头、中军、趋势核心和前排强势识别，不作为板块资金的主来源。
- 今日板块资金曲线优先使用公开源提供的板块分时资金流；如果只拿到实时快照，系统在交易时段内采样沉淀曲线；最后才使用估算曲线兜底。
- 题材雷达与资金流证据使用 5 分钟前后端缓存，降低切换页面和重复访问时的外部行情请求；手动刷新通过 `force_refresh=true` 绕过缓存。
- 信息差资讯通过 API 获取，不能替代买入信号；资讯必须经过板块映射、资金流验证和买入检查。
- 个人交易理由、计划、退出卡和复盘属于主观纪律数据，由用户录入。
- 不把生产数据打进镜像。数据库文件通过 Docker volume 保存。

## 风控原则

- 标准短线默认单票 40% 上限，持仓级止损 4%。
- 集中进攻模式允许更高仓位，但超过 60% 前必须填写退出卡。
- 规则引擎优先输出“可买 / 等确认 / 不买”，避免只给分数不拦截。
- 信息差模块只输出“值得跟踪 / 等确认 / 谨慎”，不直接输出“买入”。
